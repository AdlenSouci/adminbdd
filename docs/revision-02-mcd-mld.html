Du MCD au MLD — Méthodologie MERISE

Tout le cours PDF « Méthodologie de modélisation de base de données » — aucune notion omise.

Rappel MERISE — 3 étapes

  • Analyse : comprendre le domaine, identifier entités / attributs / relations → produit le MCD.
  • Conception : transformer le MCD en modèle logique (normalisation, règles) → produit le MLD.
  • Réalisation : traduire le MLD en schéma relationnel SQL et implémenter dans un langage.
  • Processus itératif et structuré ; chaque étape a des objectifs et des livrables définis.
  • Objectif principal : séparer données et traitements pour une modélisation claire et flexible.
MERISE jette un pont entre les besoins des utilisateurs et les solutions techniques des informaticiens.

Étape 1 — Analyse conceptuelle

Objectif

  • Comprendre le domaine et identifier les éléments clés : entités, attributs, relations, processus.
  • Cette étape pose les fondations du modèle de données.

Techniques de recueil des besoins

  • Entretiens : discussions avec les utilisateurs pour comprendre besoins et processus quotidiens.
  • Questionnaires : recueil quantitatif et qualitatif auprès de nombreux utilisateurs.
  • Analyse de documents : étude des procédures, rapports, documents existants.

Identification des entités et attributs

  • Exemple université :
EntitéAttributs
ÉtudiantMatricule, Nom, Date de Naissance, Adresse
CoursCode, Titre, Nombre de Crédits
ProfesseurID, Nom, Spécialité

Identification des relations

  • Un étudiant suit un cours (0,N — 0,N via suit).
  • Un professeur enseigne un cours (1,1 — 0,N via enseigne).
  • Un cours est dispensé dans une salle.

Création du MCD

  • Représentation visuelle : rectangles = entités, losanges = associations, lignes = liens.
  • Vue conceptuelle, indépendante des considérations techniques.

Outils de modélisation mentionnés

  • MOCODO.net : logiciel open source MERISE, génère du code SQL.
  • Looping : outil en ligne MCD + génération SQL.

Étape 2 — Conception logique (MCD → MLD)

  • Appliquer règles de normalisation et d'optimisation pour obtenir un MLD.
  • Les entités MCD deviennent des tables.
  • Les attributs sont organisés selon les formes normales.
  • Les relations sont transformées en contraintes d'intégrité référentielle (clés étrangères).
  • On utilise le GDF (Graphe des Dépendances Fonctionnelles) comme outil de transformation.

Règle 1 — Toute entité → une table

  • Chaque entité du MCD devient une table dans le MLD.
  • L'identifiant de l'entité devient la clé primaire de la table.
  • Les attributs deviennent les colonnes.

Règle 2 — Relation 1:1 (One-To-One)

  • Cardinalités maximales : x,1 des deux côtés.
  • Option A — Fusionner les tables si la liaison est systématique (plus simple).
  • Option B — Ne pas fusionner : la clé primaire de l'entité forte devient clé étrangère dans l'entité faible et sert aussi de nouvelle clé primaire à cette entité.
  • Exemple du cours : user (0,1) — profile (1,1) via has.
Ce MCD peut se transformer de deux façons selon le besoin de séparation.

Règle 3 — Relation 1:N (One-To-Many)

  • Cardinalités maximales : x,N d'un côté et x,1 de l'autre.
  • La clé primaire de l'entité ayant la cardinalité maximale N devient une clé étrangère dans l'entité ayant la cardinalité maximale 1.
  • Exemple du cours : pays (1,N) — ville (1,1) via possede.
  • Résultat MLD : la table ville contient une colonne code_pays (FK vers pays).
À retenir : la FK se place du côté max = 1 (ville), elle référence la table côté max = N (pays).

Règle 4 — Relation N:N (Many-To-Many)

  • Cardinalités maximales N des deux côtés.
  • L'association devient une table d'association (aussi appelée table pivot ou table de jointure).
  • La clé primaire de cette table est composée des deux clés étrangères des tables qu'elle associe.
  • Exemple du cours : acteur (1,N) — film (1,N) via jouer avec un attribut role.
  • Résultat MLD : table jouer(acteur_id FK, film_id FK, role) avec PK composée (acteur_id, film_id).

Résumé des 4 règles

Cardinalité maxTypeRésultat MLD
1 des deux côtésOne-To-OneFusion ou FK = PK côté faible
N d'un côté, 1 de l'autreOne-To-ManyFK dans la table côté max=1
N des deux côtésMany-To-ManyTable d'association (PK composée)
