Droits, Utilisateurs & Rôles — MySQL / MariaDB

Tout le cours (gestion_droit.txt) — CREATE USER, GRANT, REVOKE, rôles, exercice corrigé.

Utilisateur BDD ≠ utilisateur applicatif

  • Un utilisateur BDD = compte pour se connecter au serveur MySQL.
  • Un utilisateur applicatif = personne qui s'inscrit sur votre site web (ex. adhérent dans la table biblio).
  • L'appli utilise un compte BDD dédié pour se connecter (pas le root !) :
// PHP/PDO
$db = new PDO('mysql:host=localhost;dbname=mabase', 'mon_user', 'password');
Par défaut WAMP : root sans mot de passe → interdit en production !

Voir les utilisateurs (MySQL/MariaDB seulement)

SELECT user, host, password FROM mysql.user;

CREATE USER

CREATE USER [IF NOT EXISTS] 'nom'@'hote' IDENTIFIED BY 'password';

-- Authentification par socket UNIX (délégation OS)
CREATE USER 'www-data'@'localhost' IDENTIFIED VIA unix_socket;

Valeurs possibles pour hote

  • localhost — connexion locale (cas le plus courant)
  • Adresse IP exacte — ex. '192.168.1.10'
  • Nom de domaine — ex. 'mon-serveur.example.com'
  • % — tout hôte/toute IP (attention sécurité)
  • '192.168.0.%' — sous-réseau complet
  • '%.nomdomaine.org' — tout sous-domaine
Même nom d'utilisateur + hôtes différents = comptes DISTINCTS en MySQL.

Autres opérations sur un utilisateur

DROP USER [IF EXISTS] 'nom'@'hote';
ALTER USER 'nom'@'hote' IDENTIFIED BY 'new_password';

Plugins d'authentification

  • unix_socket : l'utilisateur système est authentifié par sa socket (sans mot de passe BDD).
  • LDAP : délégation à un annuaire LDAP.

Les droits disponibles

Droits DML courants

  • SELECT · INSERT · UPDATE · DELETE

Droits DDL et objets

  • Création : CREATE (tables, vues, routines, bases, utilisateurs)
  • Suppression : DROP
  • Exécution de routines : EXECUTE

Droits d'administration

  • SHUTDOWN : arrêter le serveur
  • RELOAD : recharger les tables de privilèges
  • CREATE USER : gérer les comptes

GRANT — Attribuer des droits

-- Droits sur une table
GRANT SELECT ON test.livre TO 'user1'@'localhost';

-- Droits avec possibilité de les transmettre
GRANT SELECT ON test.livre TO 'user1'@'localhost' WITH GRANT OPTION;

-- Droits sur toute une base
GRANT ALL ON web.* TO 'www-data'@'localhost';

-- Droits sur des colonnes précises
GRANT SELECT, INSERT(title), UPDATE(title) ON test.livre TO 'user2'@'localhost';

WITH GRANT OPTION

  • Permet à l'utilisateur de transmettre les droits qu'il a reçus à d'autres utilisateurs.
  • Un utilisateur PEUT transmettre ses droits même sans la capacité de créer un utilisateur : oui.
  • Mais il ne peut transmettre que les droits qu'il possède lui-même (pas plus).

REVOKE — Révoquer des droits

REVOKE SELECT ON test.livre FROM 'user2'@'localhost';

-- Révoquer sur des colonnes
REVOKE INSERT(description), UPDATE(description) ON test.livre FROM 'user2'@'localhost';
Deux points importants :
1. user2 garde les droits jusqu'à la fin de sa session courante.
2. La révocation n'est pas en cascade : si user1→user2→user3, révoquer user2 ne retire pas les droits de user3.

Portée des droits

  • Global (*.*) : s'applique à toutes les bases.
  • Table (ma_base.ma_table) : une table précise.
  • Colonne (INSERT(col1)) : une ou plusieurs colonnes.

SHOW GRANTS — Vérifier les droits

SHOW GRANTS FOR 'user2'@'localhost';
-- Résultat exemple :
-- GRANT USAGE ON *.* TO `user2`@`localhost`
-- GRANT SELECT ON `test`.* TO `user2`@`localhost` WITH GRANT OPTION
-- GRANT SELECT, INSERT (title), UPDATE (title) ON `test`.`livre` TO `user2`@`localhost`

Technique : restreindre par colonne (2 approches)

Exercice type du cours

user2 peut lire toutes les colonnes de livre, insérer et modifier le titre, mais pas la description.

Approche directe (peu de colonnes)

GRANT SELECT, INSERT(title), UPDATE(title) ON test.livre TO 'user2'@'localhost';

Approche inverse (beaucoup de colonnes)

-- 1. Accorder sur toutes les colonnes
GRANT SELECT, INSERT, UPDATE ON test.livre TO 'user2'@'localhost';
-- 2. Restreindre les colonnes sensibles
REVOKE INSERT(description), UPDATE(description) ON test.livre FROM 'user2'@'localhost';

Les rôles (MySQL 8+ / MariaDB 10+)

  • Principe : créer un rôle → lui affecter des droits → assigner le rôle à un utilisateur.
  • Rôles imbriqués : un rôle peut inclure un autre rôle (hiérarchies complexes).
-- Créer un rôle
CREATE ROLE 'mon_role';

-- Donner des droits au rôle
GRANT SELECT, INSERT ON ma_base.* TO 'mon_role';

-- Assigner le rôle à un utilisateur
GRANT 'mon_role' TO 'toto'@'localhost';

-- Rôles imbriqués
GRANT 'role_employe' TO 'role_manager';  -- manager hérite des droits employé

-- Activer un rôle manuellement
SET ROLE 'mon_role';

-- Activer automatiquement à la connexion
SET DEFAULT ROLE 'mon_role' FOR 'toto'@'localhost';

-- Révoquer un rôle d'un utilisateur
REVOKE 'mon_role' FROM 'toto'@'localhost';

-- Supprimer un rôle
DROP ROLE 'mon_role';
Un rôle attribué n'est actif qu'après activation (SET ROLE ou SET DEFAULT ROLE).

Exemple pratique complet (cours)

-- Étape 1 : Créer les utilisateurs
CREATE USER 'alice'@'localhost' IDENTIFIED BY 'password';
CREATE USER 'bob'@'localhost' IDENTIFIED BY 'password';

-- Étape 2 : Créer les rôles
CREATE ROLE 'lecteur';
CREATE ROLE 'editeur';

-- Étape 3 : Attribuer les privilèges aux rôles
GRANT SELECT ON ma_base.* TO 'lecteur';
GRANT SELECT, INSERT, UPDATE ON ma_base.* TO 'editeur';

-- Étape 4 : Assigner les rôles
GRANT 'lecteur' TO 'alice'@'localhost';
GRANT 'editeur' TO 'bob'@'localhost';

-- Étape 5 : Activer automatiquement à la connexion
SET DEFAULT ROLE 'lecteur' FOR 'alice'@'localhost';
SET DEFAULT ROLE 'editeur' FOR 'bob'@'localhost';

-- Étape 6 : Vérifier
SHOW GRANTS FOR 'alice'@'localhost';
SHOW GRANTS FOR 'bob'@'localhost';

Exercice corrigé — Gestion des rôles (cours)

Énoncé : base emptemp (employees, departments, salaries)

  • admin : tous les privilèges sur toutes les tables.
  • analyste : SELECT sur toutes les tables.
  • gestionnaire : SELECT + UPDATE sur employees et departments.
-- Créer la base et les tables
CREATE DATABASE IF NOT EXISTS emptemp; USE emptemp;
CREATE TABLE departments(id INT AUTO_INCREMENT PRIMARY KEY, name VARCHAR(255) UNIQUE);
CREATE TABLE employees(id INT AUTO_INCREMENT PRIMARY KEY, first_name VARCHAR(255),
  last_name VARCHAR(255), department_id INT NOT NULL,
  FOREIGN KEY (department_id) REFERENCES departments(id));
CREATE TABLE salaries(id INT AUTO_INCREMENT PRIMARY KEY, amount DECIMAL(10,2),
  from_date DATE DEFAULT CURRENT_DATE, to_date DATE DEFAULT '9999-01-01',
  employee_id INT NOT NULL, FOREIGN KEY (employee_id) REFERENCES departments(id));

-- Données de test
INSERT INTO departments(name) VALUES ('IT'), ('RH');
INSERT INTO employees(first_name, last_name, department_id)
  VALUES ('toto','leharicot',(SELECT id FROM departments WHERE name='IT')),
         ('titi','letirlibibi',(SELECT id FROM departments WHERE name='RH'));

-- Créer les rôles
CREATE ROLE admin; CREATE ROLE analyste; CREATE ROLE gestionnaire;

-- Attribuer les privilèges
GRANT ALL ON emptemp.* TO admin;
GRANT SELECT ON emptemp.* TO analyste;
GRANT SELECT, UPDATE ON emptemp.employees TO gestionnaire;
GRANT SELECT, UPDATE ON emptemp.departments TO gestionnaire;

-- Créer les utilisateurs
CREATE USER admin_user@localhost IDENTIFIED BY 'securepass';
CREATE USER data_user@localhost IDENTIFIED BY 'analyzepass';
CREATE USER manager_user@localhost IDENTIFIED BY 'managerpass';

-- Affecter les rôles
GRANT admin TO admin_user@localhost;
GRANT analyste TO data_user@localhost;
GRANT gestionnaire TO manager_user@localhost;

-- Activer les rôles par défaut
SET DEFAULT ROLE admin FOR admin_user@localhost;
SET DEFAULT ROLE analyste FOR data_user@localhost;
SET DEFAULT ROLE gestionnaire FOR manager_user@localhost;

-- Tests
mysql -u admin_user -p emptemp;
SELECT * FROM employees;
UPDATE departments SET name = 'HR' WHERE name = 'RH';
