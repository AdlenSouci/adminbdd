Dans MySQL (et dans les autres SGBD), il est possible de créer des nouveaux utilisateurs de la base de données, et de configurer leurs droits. Il est aussi possible de créer des rôles et d'affecter des rôles aux utilisateurs.

⚠️ un utilisateur de base de données n'est pas forcément un utilisateur au sens « applicatif », c'est-à-dire un utilisateur qui peut s'enregistrer et de se connecter à votre application. Exemple : la table adhérent correspond à des utilisateurs de l'application biblio.

On a donc un utilisateur qui se connecte par le biais de formulaire (Web par exemple). Pour vérifier la connexion de cet utilisateur, le code PHP doit pouvoir se connecter à la base de données. Pour cela, le code utilise un utilisateur de base de données dédié.

Exemple de snippet de connexion donné pour une installation WAMP configurée par défaut :

$connexion = new PDO('mysql:host=localhost;dbname=mabase', 'root', ''); // Par défaut WAMP ne met pas de mot de passe à l'utilisateur de base de données root (et c'est pas bien !!!!)
Pour voir les utilisateurs de base de données présents dans une base : (MySQL et MariaDb only)

SELECT user, host, password
FROM mysql.user;
Concepts de base des privilèges MySQL
Les privilèges (en fait les droits, ce sont des termes équivalents) déterminent ce qu’un utilisateur peut faire dans une base de données. Ces privilèges s’appliquent à différents niveaux :

Global : Affecte toutes les bases de données. Exemples : CREATE USER, RELOAD.
Base de données : Affecte une base spécifique. Exemples : CREATE, SELECT, DROP.
Table : Affecte des actions sur une table particulière. Exemples : INSERT, UPDATE.
Colonnes : Permet des actions spécifiques sur des colonnes.
Routines : Pour les procédures et fonctions stockées.
Création d'utilisateur de base de données
CREATE USER [IF NOT EXISTS] `nom_utilisateur`@`hote` IDENTIFIED BY 'password'; -- On peut aussi identifier l'utilisateur par le biais d'une socket UNIX : IDENTIFIED VIA unix_socket;
hote peut prendre comme valeur :

localhost (cas le plus courant)
une adresse IP
un nom de domaine
% => tout domaine/toute ip
On peut utiliser le caractère % pour indiquer :

un sous réseau : '192.168.0.%'

n'importe quel sous domaine : '%.nomdomaine.org'
hote correspond à l'adresse IP depuis laquelle l'utilisateur se connecte. On peut donc avoir plusieurs fois le même utilisateur avec des hôtes différents.

Les manipulations possibles sur un utilisateur :

la suppression : DROP USER [IF EXISTS] ...

la modification : ALTER USER ....
la création : CREATE USER ...
Il existe des plugins d'authentification :

unix_socket : l'utilisateur (standard ou système) qui exécute un script SQL voit sont authentification *déléguée au système par sa socket d'authentification.
LDAP :
Exemple d'utilisation de socket UNIX : donner les droits sur une base à l'utilisateur www-data (qui correspond à apache ou nginx) :

CREATE USER 'www-data'@'localhost' IDENTIFIED VIA unix_socket;
CREATE DATABASE web;
GRANT ALL ON web.* TO 'www-data'@'localhost';
Un peu de remplissage :

CREATE TABLE test(id INT PRIMARY KEY AUTO_INCREMENT, name VARCHAR(255));
INSERT INTO test(name) VALUES('Ceci est un test');
On peut maintenant utiliser l'utilisateur www-data pour se connecter à la base de données web dans un script php qui sera exécuté par apache (www-data) :

<?php

// On demande à php d'afficher les erreurs sur la page (attention, ne pas faire cela en production
ini_set('display_errors', 1);
ini_set('display_startup_errors', 1);
error_reporting(E_ALL);

try { // On se connecte à la base de données avec l'utilisateur www-data $db = new PDO('mysql:host=localhost;dbname=web', 'www-data', ''); // On indique au driver que s'il rencontre une erreur, il doit lever une execption. $db->setAttribute(PDO::ATTR_ERRMODE, PDO::ERRMODE_EXCEPTION);
// On crée une requête qui récupère toutes les données de la table test $rqt = "select * from test"; // On execute la requête $stmt = $db->query($rqt); // fetchAll permet de récupérer l'ensemble des résultats de la requête. print_r($stmt->fetchAll()); // On "dump" les résultats à l'écran } catch (Exception $e) { // Si on rencontre une exception (par exemple une erreur de connexion, on affiche le problème rencontré). echo "echec ". $e->getMessage(); // Ne pas afficher le message "directement" en production } Si l'on place ce fichier que l'on nomme test-db.php dans le répertoire /var/www/html, celui-ci sera exécuté par apache (dans sa configuration par défaut). On peut le tester se rendant sur l'url http://localhost/test-db.php

sudo chown www-data /var/www/html/test-db.php # On donne la propriété du fichier à www-data
On peut aussi tester le script directement avec l'utilisateur local :

php /var/www/html/test-db.php
On obtient l'erreur suivante (car l'utilisateur courant n'est pas www-data) :

echec SQLSTATE[HY000] [1698] Access denied for user 'www-data'@'localhost'
⚠️ ce n'est pas parcequ'il est possible d'attribuer les droits à l'utilisateur www-data que c'est une bonne idée de le faire !!!!

Les droits

Créer des utilisateurs c'est bien, mais souvent, on en crée des nouveaux pour leur donner des droits particuliers.

Les droits courants sur les opérations de manipulation (DML) :

SELECT

INSERT
UPDATE
DELETE
Ces droits s'appliquent sur des tables, mais on peut aussi restreindre aux colonnes.

Transmission des droits.
Si l'on donne des droits à un utilisateur avec l'instruction WITH GRANT OPTION, cet utilisateur peut transmettre les droits dont il dispose à un autre utilisateur.

Un utilisateur peut-il transmettre ses droits à un autre s'il n'a pas la possibilité de créer un utilisateur ?

oui

Un exemple : on se connecte en root à mysql pour créer 2 utilisateurs user1 et user2, ainsi qu'une base test qui contiendra la table livre. user1 aura la possibilité de transmettre ses droits.

CREATE USER user1@localhost IDENTIFIED BY 'password';
CREATE USER user2@localhost IDENTIFIED BY 'password';

CREATE DATABASE IF NOT EXISTS test; USE test; -- on se positionne sur la base test; -- On crée une table CREATE TABLE livre ( id INT PRIMARY KEY AUTO_INCREMENT, title VARCHAR(255), description TEXT ); -- On insère une ligne INSERT INTO livre (title) VALUES("Le SQL c'est génial (ou presque)");

-- On donne les droits SELECT à user1 sur la table test.livre
GRANT SELECT ON test.livre TO user1@localhost WITH GRANT OPTION;

On se connecte avec user1 : mysql -u user1 -p. On essaye de jouer avec les droits de user1 en testant :

l'utilisation d'une base qui n'est pas test. => impossible

l'utilisation de la base test. => ok
la sélection des données de test.livre => ok
l'insertion de données dans test.livre => impossible
la transmission des droits SELECT à user2 => ok
la transmission des droits INSERT à user2 => impossible
USE web; -- ERROR 1044 (42000): Access denied for user 'test'@'localhost' to database 'web'
USE test; -- Ok
SELECT * FROM livre; -- > ok
INSERT INTO livre(title) VALUES ('Insertion impossible'); -- ERROR 1142 (42000): INSERT command denied to user 'test'@'localhost' for table 'livre'
GRANT SELECT ON test.livre TO user2@localhost; --> ok
GRANT INSERT ON test.livre TO user2@localhost; --> ERROR 1044 (42000): Access denied for user 'test'@'localhost' to database 'test';
On vérifie que user2 a bien les droits en lecture sur test.livre : mysql -D test -u user2 -p

SELECT * FROM livre; -- > ok
Révocation des droits
La révocation de droit consiste à enlever des droits à des utilisateurs. La syntaxe est très similaire au GRANT :

REVOKE SELECT ON test.livre FROM user2@localhost;
⚠️ user2 garde les droits sur la durée de session.

La révocation ne s'effectue pas en cascade : user1 donne les droits à user2 qui les donne à son tour à user3. Si on revoque les droits de user2, user3 garde les siens.

la portée des droits.

Certains droits ont différentes portées :

global => toute les base
table => au niveau d'une table
column => au niveau d'une colonne.
Exercice : Faites en sorte que user2 puisse lire les colonnes de livre, insérer un titre et le modifier, mais pas supprimer de livre. Il ne pourra pas insérer de données dans la colonne description, ni la modifier.

Version de Tristan (ok, car pas beaucoup de colonnes)

GRANT select, insert(`title`), update(`title`) ON test.livre to user2@localhost;
Si on a beaucoup de colonnes et que l'on veut restreindre sur un petit nombre, on peut accorder les droits sur toutes les colonnes et restreindre par la suite :

GRANT select, insert, update ON test.livre to user2@localhost;
REVOKE insert(`description`), update(`description`) ON test.livre FROM user2@localhost;
Vérification des droits des utilisateurs
On peut vérifier les droits d'un utilisateur avec la commande suivante :

SHOW GRANTS FOR user2@locahost;
+---------------------------------------------------------------------------------------+
| Grants for user2@localhost |
+---------------------------------------------------------------------------------------+
| GRANT USAGE ON *.* TO `user2`@`localhost` |
| GRANT SELECT ON `test`.* TO `user2`@`localhost` WITH GRANT OPTION |
| GRANT SELECT, INSERT (title), UPDATE (title) ON `test`.`livre` TO `user2`@`localhost` |
+---------------------------------------------------------------------------------------+
Les rôles
Depuis MySQL 8.0 / Maria 10 (me semble-t-il), il est possible de définir des rôles utilisateurs. Le principe est le suivant :

on crée un rôle et on lui affecte la politique de droit adéquate.
on crée un utilisateur et on lui assigne un rôle.
La syntaxe :

CREATE ROLE 'mon_role';
GRANT liste_droit TO db.* TO 'mon_role';
On a donné une liste de droit au rôle, maintenant on donne les droits du rôle à l'utilisateur :

GRANT 'mon_role' TO 'toto'@'localhost', 'titi'@'localhost';
Il est aussi possible d'imbriquer les rôles (c'est à dire créer un rôle depuis un autre rôle), ce qui permet de gérer des droits dans des hiérachies complèxe. Par exemple, nn rôle manager pourrait inclure le rôle employé pour hériter de ses privilèges.

GRANT 'role_employe' TO 'role_manager';
Activer un rôle
Un rôle attribué n’est actif qu’après activation. Vous pouvez activer un rôle avec :

SET ROLE 'nom_role';
Exemple :

SET ROLE 'analyste';
Pour activer automatiquement un rôle lorsqu’un utilisateur se connecte :

SET DEFAULT ROLE 'nom_role' FOR 'utilisateur'@'hote';
Révoquer un rôle
Pour retirer un rôle d’un utilisateur :

REVOKE 'nom_role' FROM 'utilisateur'@'hote';
Supprimer un rôle
Si un rôle n’est plus nécessaire, vous pouvez le supprimer :

DROP ROLE 'nom_role';
Exemple pratique complet
Étape 1 : Création des utilisateurs
Créez deux utilisateurs :

CREATE USER 'alice'@'localhost' IDENTIFIED BY 'password';
CREATE USER 'bob'@'localhost' IDENTIFIED BY 'password';
Étape 2 : Création des rôles
Créez deux rôles :

CREATE ROLE 'lecteur';
CREATE ROLE 'editeur';
Étape 3 : Attribution des privilèges aux rôles
Donnez les privilèges suivants aux rôles :

Le rôle lecteur peut lire les données d’une base de données.
Le rôle editeur peut lire et modifier les données.
GRANT SELECT ON ma_base.* TO 'lecteur';
GRANT SELECT, INSERT, UPDATE ON ma_base.* TO 'editeur';
Étape 4 : Assigner des rôles aux utilisateurs
Attribuez les rôles :

GRANT 'lecteur' TO 'lucie'@'localhost';
GRANT 'editeur' TO 'paul'@'localhost';
Étape 5 : Activer les rôles par défaut
Configurez les rôles pour qu’ils soient activés automatiquement :

SET DEFAULT ROLE 'lecteur' FOR 'alice'@'localhost';
SET DEFAULT ROLE 'editeur' FOR 'bob'@'localhost';
Étape 6 : Vérifier les privilèges
Connectez-vous en tant que alice ou bob et vérifiez les droits actifs :

SHOW GRANTS FOR 'alice'@'localhost';
SHOW GRANTS FOR 'bob'@'localhost';
4. Exercice : Gestion des rôles
Objectif :
Créer et utiliser des rôles pour une base de données d’entreprise contenant les tables suivantes : employees, departments, et salaries.

Instructions :
Créez les rôles suivants :

admin : tous les privilèges sur toutes les tables.
analyste : droit de lecture (SELECT) sur toutes les tables.
gestionnaire : droit de lecture et de modification (SELECT, UPDATE) sur les tables employees et departments.
Associez les rôles à ces utilisateurs :

admin pour l’utilisateur admin_user (mot de passe : securepass).
analyste pour l’utilisateur data_user (mot de passe : analyzepass).
gestionnaire pour l’utilisateur manager_user (mot de passe : managerpass).
Activez les rôles par défaut et testez les privilèges en exécutant les commandes suivantes pour chaque utilisateur :

Lire les données de la table employees.
Modifier un enregistrement dans la table departments.
Correction
Creation de la base emptemp :
CREATE DATABASE IF NOT EXISTS emptemp;
USE emptemp;

CREATE TABLE departments(
id INT AUTO_INCREMENT PRIMARY KEY,
name VARCHAR(255) UNIQUE
);

CREATE TABLE employees(
id INT AUTO_INCREMENT PRIMARY KEY,
first_name VARCHAR(255),
last_name VARCHAR(255),
department_id INT NOT NULL,
FOREIGN KEY (department_id) REFERENCES departments(id)
);

CREATE TABLE salaries(
id INT AUTO_INCREMENT PRIMARY KEY,
amount DECIMAL(10,2) ,
from_date DATE DEFAULT CURRENT_DATE,
to_date DATE DEFAULT '9999-01-01',
employee_id INT NOT NULL,
FOREIGN KEY (employee_id) REFERENCES departments(id)
);

INSERT INTO departments(name) VALUES ('IT'), ('RH');
INSERT INTO employees(first_name, last_name, department_id) VALUES ('toto', 'leharicot', (SELECT id FROM departments WHERE name = 'IT')), ('titi', 'letirlibibi', (SELECT id FROM departments WHERE name = 'RH'));

Création des rôles :

CREATE ROLE admin;
CREATE ROLE analyste;
CREATE ROLE gestionnaire;

-- Gestion des privilèges pour les rôles :

GRANT ALL on emptemp.* TO admin;
GRANT SELECT on emptemp.* TO analyste;
GRANT SELECT, UPDATE on emptemp.employees TO gestionnaire;
GRANT SELECT, UPDATE on emptemp.departments TO gestionnaire;
Association des rôles et des utilisateurs :
-- Création des utilisateurs au préalable
CREATE USER admin_user@localhost IDENTIFIED BY 'securepass';
CREATE USER data_user@localhost IDENTIFIED BY 'analyzepass';
CREATE USER manager_user@localhost IDENTIFIED BY 'managerpass';

-- Affectation des rôles :
GRANT admin TO admin_user@localhost;
GRANT analyste TO data_user@localhost;
GRANT gestionnaire TO manager_user@localhost;
Activation des rôles :
SET DEFAULT ROLE admin FOR admin_user@localhost;
SET DEFAULT ROLE analyste FOR data_user@localhost;
SET DEFAULT ROLE gestionnaire FOR manager_user@localhost;
Test pour chaque utilisateur :

mysql -u admin_user -p emptemp;

SELECT * FROM employees;
UPDATE departments SET name = 'HR' WHERE name = 'RH';
mysql -u data_user -p emptemp;
mysql -u manager_user -p emptemp;
