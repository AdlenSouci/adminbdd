MCD — MERISE

Points à retenir alignés sur le cours PDF « Modèle conceptuel de données ».

Objectifs

  • Situer le MCD dans MERISE.
  • Définir entité, attribut, association, identifiant, cardinalités.
  • Lire / critiquer un schéma (dont ternaire et réflexive).
  • Préparer le passage au MLD (surtout via les cardinalités).

Plan du cours (rappel)

  • MCD · MERISE · entités · attributs · associations
  • Identifiants · clés candidates / primaire · cardinalités
  • Exemples · impact MLD · ternaires · réflexives · cas bibliothèque

MERISE — trois niveaux

NiveauQuestionModèle
ConceptuelQuelles données ?MCD
LogiqueComment les organiser ?MLD
PhysiqueQuel SGBD / implémentation ?MPD
  • Méthode française, années 1970 ; données modélisées à part des traitements.

MCD, MLD, MPD

  • MCD : quelles informations le SI doit connaître (sans tables SQL).
  • MLD : tables, clés, normalisation.
  • MPD : types, index, contraintes physiques.
  • Le MCD ignore les contraintes techniques : il doit rester lisible métier.

Entité

  • Objet du réel dont on garde des infos ; existence propre ; occurrences aux mêmes propriétés.
  • Nom : singulier, explicite, pas d’abréviation floue.
  • Ex. cours : CLIENT, PRODUIT, COMMANDE avec attributs typiques (n°, nom, prix…).
  • Éviter : pluriel (CLIENTS), sigles obscurs (CLI), préfixes type TBL_.

Attributs

  • Simple : une valeur atomique.
  • Calculé : dérivé (ex. âge depuis date de naissance).
  • Composition : ex. n° sécu — attention à la granularité en conception.
  • Viser l’atomicité pour faciliter la 1NF au MLD.

Associations

  • Lien sémantique entre entités.
  • Binaire (2), ternaire (3), n-aire, réflexive (même entité).
  • Ex. CONTENIR COMMANDE–PRODUIT avec quantite sur l’association.
  • Réflexive ex. SUPERVISER sur EMPLOYE : hiérarchie, sens à préciser.

Identifiant

  • Unicité : une occurrence = une valeur d’identifiant.
  • Non-nullité obligatoire.
  • Stabilité dans le temps.
  • Minimalité : aucun sous-ensemble d’attributs ne suffit seul.
  • Ex. numero_etudiant oui ; nom + prenom non (homonymes, changements).

Clés candidates et clé primaire

  • Candidats : tous les identifiants possibles pour l’entité.
  • Primaire : le candidat choisi (souvent noté #).
  • Choix : stabilité, simplicité, perf future, sens métier.
  • email = candidat seulement si garanti unique à vie.

Cardinalités (min, max)

  • 0 optionnel · 1 obligatoire · N plusieurs.
  • 0,1 au plus une fois · 1,1 exactement une fois.
  • 0,N zéro ou plusieurs · 1,N au moins une fois.

Lecture (ex. client — commande)

  • Côté client « passe une commande » : souvent 1,N vers commande.
  • Côté commande : 1,1 vers un seul client.
  • Toujours valider avec le métier (un client sans commande ?).

Exemples cardinalités

  • EMPLOYE — DEPARTEMENT (APPARTENIR) : un employé dans 1 dept ; un dept a 1,N employés.
  • PRODUIT — COMMANDE (CONTENIR) : produit jamais commandé → souvent 0,N ; lignes de commande selon règle métier.

MCD → MLD (règle binaire)

  • Ex. POSSEDER : PERSONNE (1,N) — VOITURE (0,1).
  • Une personne possède 1 à N voitures ; une voiture : 0 ou 1 propriétaire.
À retenir : côté cardinalité max = N (0,N ou 1,N) vs côté max = 1 (0,1 ou 1,1) → l’identifiant de l’entité « côté N » est recopié dans l’entité « côté 1 » : clé étrangère dans la table du côté 1 (ex. FK personne dans VOITURE).

Ternaire

  • Trois entités liées en même temps ; une décomposition binaire perd le sens.
  • Ex. participation : JOUEUR + EQUIPE + MATCH.
  • Réserver aux cas nécessaires (complexité).

Réflexive

  • Une entité reliée à elle-même (hiérarchie, catégories, etc.).
  • Clarifier le sens des deux rôles (superviseur / supervisé).

Cas bibliothèque (résumé)

  • Entités de base : LIVRE, AUTEUR, ADHERENT (+ attributs ISBN, dates…).
  • ECRIRE LIVRE–AUTEUR (souvent N,N). EMPRUNTER ADHERENT–LIVRE avec dates.
  • Questions : un seul exemplaire ? historique ? → entité EXEMPLAIRE, lien avec LIVRE.
  • Catégories : association LIVRE–CATEGORIE ; schéma final du cours à recoller mentalement.

« Normalisation » au stade MCD

  • Moins de redondance, attributs cohérents avec la clé (esprit 2NF/3NF au MLD).

Identifiant technique vs métier

  • Seul le code-barres = risque si règles externes changent.
  • Cours : souvent id_… stable + garder code-barres en attribut.

Erreurs fréquentes

  • Confondre entité et attribut (ex. adresse → parfois entité ADRESSE + HABITER).
  • Cardinalités non validées avec le métier.
  • Identifiants instables (email, téléphone, nom seul).

Contrôle final

  • Identifiants stables ? Cardinalités cohérentes ? Pas de redondance inutile ?
  • Règle d’or : le MCD doit être compréhensible par les utilisateurs métier.
