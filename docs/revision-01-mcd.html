3. Qu’est-ce que MERISE ?

Méthode française de conception de systèmes d’information, développée dans les années 1970.

Trois niveaux d’abstraction :

NiveauQuestionModèle
ConceptuelQue fait-on ? (quelles données ?)MCD
LogiqueComment organise-t-on les données ?MLD
PhysiqueAvec quels outils / SGBD ?MPD

Séparation données / traitements : MERISE modélise les données (MCD, MLD, MPD) à part des traitements (MCT, MOT, etc., hors fiche).

4. Place du MCD dans MERISE

  • MCD : quelles informations le système doit connaître ?
  • MLD : comment les organiser (tables, clés, normalisation) ?
  • MPD : avec quoi les implémenter (SQL, types, index…) ?

Le MCD décrit la structure des informations indépendamment des contraintes techniques (pas encore de tables SQL).

5. Les entités

Définition : objet du monde réel sur lequel on conserve des informations, avec une existence propre, identifiable, et des occurrences aux mêmes propriétés.

Règles de nommage : nom au singulier, explicite « métier », sans abréviation ambiguë.

Exemples d’entités (cours) :

CLIENT — numero_client, nom, prenom, email PRODUIT — code_produit, libelle, prix, stock COMMANDE — numero_commande, date_commande, montant_total

✅ Bon : CLIENT, PRODUIT · ❌ À éviter : CLIENTS, CLI, TBL_CLIENT

6. Les attributs

Types (cours) :

  • Simple : non décomposable (une valeur atomique).
  • Calculé : dérivé d’autres attributs (ex. age à partir de date_naissance).
  • Composition implicite : ex. numero_secu (genre, année, mois de naissance, …) — attention en conception.

💡 Conseil : privilégier des attributs atomiques dans le MCD pour faciliter la 1NF au niveau logique.

7. Les associations

Définition : lien sémantique entre deux ou plusieurs entités.

Types : binaire (2 entités), ternaire (3), n-aire (n), réflexive (une même entité reliée à elle-même).

Association binaire (ex.) CONTENIR entre COMMANDE et PRODUIT avec attribut quantite ; cardinalités typiques sur le schéma du cours.

Association réflexive (ex.) SUPERVISER sur EMPLOYE : un employé supervise 0 à N employés ; un employé est supervisé par 0 ou 1 employé.

Les associations peuvent porter leurs propres attributs (ex. date sur une liaison).

8. Identifiants — concepts de base

Un identifiant est le (les) attribut(s) qui distinguent uniquement chaque occurrence.

Propriétés obligatoires :

  • 🔑 Unicité : pas deux occurrences identiques
  • ❗ Non-nullité : jamais vide
  • 📌 Stabilité : ne change pas (ou peu) dans le temps
  • ⚡ Minimalité : aucun sous-ensemble d’attributs ne suffit à identifier

Exemple (ETUDIANT) : bon identifiant numero_etudiant ; mauvais : nom + prenom (homonymes, changements).

9. Clés candidates et clé primaire

  • Clés candidates : tous les identifiants possibles pour une entité.
  • Clé primaire : la clé candidate choisie (souvent notée # sur les schémas).
  • Critères de choix : stabilité, simplicité, performance future, sens métier.

Ex. : numero_etudiant retenu ; email possible seulement si toujours unique ; le tuple (nom, prenom, date_naissance) sans garantie d’unicité = mauvais candidat.

10. Cardinalités — notation min, max

Définition : nombre minimum et maximum de participations d’une entité à une association.

  • 0 : participation optionnelle · 1 : obligatoire · N : plusieurs
  • 0,1 : au plus une fois · 1,1 : exactement une fois
  • 0,N : zéro ou plusieurs · 1,N : au moins une fois

Lecture (ex. PASSER client → commande) : un client passe au minimum 1 commande et au maximum N ; une commande est passée par 1 client et un seul.

11. Cardinalités — exemples

Employé — Département (APPARTENIR)

Un département a 1 à N employés ; un employé appartient à 1 et 1 seul département.

Produit — Commande (CONTENIR)

Questions métier : un produit peut-il ne jamais être commandé ? → souvent 0,N. Une commande peut-elle être « vide » ? → en général 1,N côté lignes de commande (selon règle métier du cours).

12. Impact des cardinalités (MCD → MLD)

Exemple POSSEDER : PERSONNE (1,N) — VOITURE (0,1). Une personne possède 1 à N voitures ; une voiture est possédée par 0 ou 1 personne.

Transformation : la table VOITURE contiendra une clé étrangère vers PERSONNE ; pas de table d’association nécessaire pour ce cas binaire.

Règle du cours (à retenir pour le MLD) : pour une association binaire avec un côté en 0,N ou 1,N et l’autre en 0,1 ou 1,1, l’identifiant de l’entité « côté N » (max = N) est recopié dans l’entité « côté 1 » (max = 1) : c’est elle qui portera la clé étrangère au niveau logique.

Ex. : clé de PERSONNE → colonne(s) FK dans VOITURE.

💡 Les cardinalités guident toute la transformation MCD → MLD.

13. Associations ternaires

Quand : trois entités liées simultanément ; impossible de décomposer sans perdre le sens.

Exemple (cours) : un joueur d’une équipe participe à un match — il faut les trois concepts (JOUEUR, EQUIPE, MATCH) pour définir PARTICIPATION.

⚠ Les ternaires sont plus complexes : ne les utiliser que si une modélisation binaire ne suffit pas.

14. Associations réflexives

Une entité liée à elle-même (ex. hiérarchie d’employés, catégories / sous-catégories, articles liés).

Précaution : bien définir le sens de la relation (qui supervise qui, lecture dans les deux sens).

15. Cas d’étude — bibliothèque

Étape 1 — Entités de base

LIVRE, AUTEUR, ADHERENT (attributs du cours : ISBN, dates, etc.). Questions : que manque-t-il ? comment lier ?

Étape 2 — Associations simples

ECRIRE entre LIVRE et AUTEUR (N,N typique). EMPRUNTER entre ADHERENT et LIVRE avec dates d’emprunt / retour. Problème soulevé : un seul exemplaire par livre ? historique des emprunts ?

Étape 3 — Enrichissement

Entité EXEMPLAIRE (état, date d’acquisition) liée à LIVRE ; catégories avec association LIVRE–CATEGORIE.

Modèle complet : regrouper LIVRE, AUTEUR, ADHERENT, EXEMPLAIRE, CATEGORIE et les associations ECRIRE, EMPRUNTER, APPARTENIR, ETRE_EXEMPLAIRE comme sur le schéma final du cours.

16. Normalisation « dans l’esprit » du MCD

Éviter redondance, assurer cohérence, faciliter la maintenance.

Règles de base : atomicité des attributs ; chaque information une fois ; attributs dépendant de la clé « entière » au niveau logique.

17. Choix des identifiants — technique vs métier

Métier seul (ex. code-barres) : risque de changement ou de règles externes.

Technique (recommandé dans le cours) : id_produit stable + conserver code_barre comme attribut.

Pourquoi l’identifiant technique : stabilité, performance, indépendance vis-à-vis des règles métier, simplicité côté système.

18. Erreurs courantes (rappel cours)

  1. Entité vs attribut : ex. adresse → parfois entité ADRESSE avec association HABITER.
  2. Cardinalités : se poser si un client peut avoir 0 commande ; valider avec le métier.
  3. Identifiants instables : email, téléphone, nom/prénom seuls — préférer un identifiant stable + garder ces données en attributs.

19. Conseils pour un MCD maintenable

Documenter les choix ; valider avec les utilisateurs ; anticiper ; rester simple ; respecter les conventions.

Vérifications finales : identifiants stables ? cardinalités cohérentes ? pas de redondance inutile ? entités et associations justifiées ?

🎯 Règle d’or (cours) : un bon MCD doit être compréhensible par les utilisateurs métier.

20. Problèmes d’identification fréquents

  • Identifiants composés trop complexes (> 3 attributs → revoir).
  • Confusion identifiant / référence (la référence = clé étrangère au MLD).
  • Oubli de la minimalité.
  • Identifiants métier trop risqués sans identifiant technique.

21. Récapitulatif — bénéfices & suite

Bénéfices d’un bon MCD : compréhension métier partagée ; base solide pour MLD/MPD ; maintenance ; performance ; cohérence des données.

Étapes suivantes (cours) : transformation MCD → MLD ; optimisation pour le SGBD ; implémentation physique.
