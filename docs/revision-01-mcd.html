Qu'est-ce que MERISE ?

  • Méthode française de conception de systèmes d'information, années 1970.
  • Séparation claire entre données et traitements.
  • 3 niveaux d'abstraction :
NiveauQuestionModèle données
ConceptuelQue fait-on ?MCD
LogiqueComment organise-t-on ?MLD
PhysiqueAvec quels outils ?MPD

Place du MCD dans MERISE

  • Le MCD répond à : « Quelles informations le système doit-il connaître ? »
  • Représente la structure des informations indépendamment de toute contrainte technique (pas de tables SQL, pas de SGBD).
  • Sert de base à la conception logique (MLD) puis physique (MPD).

Les Entités

  • Définition : objet du monde réel sur lequel on conserve des informations, avec une existence propre, identifiable, occurrences aux mêmes propriétés.
  • Règles de nommage : nom au singulier, explicite, métier, sans abréviation ambiguë.
  • Bons exemples : CLIENT, PRODUIT, COMMANDE.
  • À éviter : CLIENTS (pluriel), CLI (abréviation floue), TBL_CLIENT (préfixe technique).
  • Exemples d'attributs par entité : CLIENT → numero_client, nom, prenom, email ; PRODUIT → code_produit, libelle, prix, stock.

Les Attributs

  • Simple : non décomposable (valeur atomique) — ex. nom, prenom.
  • Calculé : dérivé d'autres attributs — ex. age depuis date_naissance.
  • Composition implicite : ex. numero_secu (genre + année + mois de naissance…).
Conseil cours : privilégier les attributs atomiques (non décomposables) dans le MCD pour respecter la 1NF au niveau logique.

Les Associations

  • Définition : lien sémantique entre deux ou plusieurs entités.
  • Types : Binaire (2 entités) · Ternaire (3 entités) · N-aire (n entités) · Réflexive (même entité).
  • Les associations peuvent porter leurs propres attributs (ex. quantite sur CONTENIR COMMANDE–PRODUIT).
  • Exemple binaire : CONTENIR (1,N — 0,N) entre COMMANDE et PRODUIT, avec attribut quantite.
  • Exemple réflexive : SUPERVISER (0,N — 0,1) sur EMPLOYE.

Identifiants — concepts de base

  • Attribut(s) qui distinguent uniquement chaque occurrence.
  • Unicité : pas deux occurrences avec le même identifiant.
  • Non-nullité : jamais vide.
  • Stabilité : ne change pas dans le temps.
  • Minimalité : aucun sous-ensemble d'attributs ne suffit seul.
  • Bon ex. : numero_etudiant (stable, unique, simple). Mauvais : nom + prenom (doublons, changements).

Clés candidates et clé primaire

  • Clés candidates : tous les identifiants possibles pour une entité.
  • Clé primaire : clé candidate choisie (notée # sur les schémas).
  • Critères de choix : stabilité · simplicité · performance future · sémantique métier.
  • Exemple sur ETUDIANT : candidats = numero_etudiant (choisi ✅) et email (seulement si toujours unique ⚠️).
  • Tuple (nom, prenom, date_naissance) = mauvais candidat (pas de garantie d'unicité).

Cardinalités — notation min,max

  • Définition : nombre minimum et maximum de participations d'une entité à une association.
  • 0 : participation optionnelle · 1 : obligatoire · N : plusieurs.
  • 0,1 : au plus une fois · 1,1 : exactement une fois · 0,N : zéro ou plusieurs · 1,N : au moins une fois.
  • Lecture exemple (PASSER entre CLIENT et COMMANDE) :
  • CLIENT (1,N) : un client passe au minimum 1 commande, au maximum N.
  • COMMANDE (1,1) : une commande est passée par 1 et 1 seul client.

Questions à se poser pour choisir

  • Un produit peut-il ne jamais être commandé ? → 0,N
  • Une commande peut-elle être vide ? → 1,N
  • Toujours valider avec le métier et documenter le choix.

Cardinalités — exemples pratiques

Employé — Département (APPARTENIR)

  • Un département a 1 à N employés.
  • Un employé appartient à 1 et 1 seul département.
  • Notation : DEPARTEMENT (1,N) — EMPLOYE (1,1).

Produit — Commande (CONTENIR)

  • PRODUIT (0,N) — COMMANDE (1,N) avec attribut quantite.

Impact des cardinalités → MLD

  • Exemple : POSSEDER — PERSONNE (1,N) — VOITURE (0,1).
  • La table VOITURE aura une clé étrangère vers PERSONNE (pas de table d'association nécessaire ici).
Règle importante du cours : cardinalité (0,1) ou (1,1) → migration d'identifiant vers cette entité (clé étrangère côté max = 1). Les cardinalités guident toute la transformation MCD → MLD.

Associations ternaires

  • Quand : 3 entités liées simultanément, impossible à décomposer sans perte de sens.
  • Exemple : PARTICIPATION liant JOUEUR + EQUIPE + MATCH (cardinalités 1,1 — 1,1 — 1,N selon le cours).
  • Pourquoi ternaire : une participation lie les trois à la fois, impossible à séparer en relations binaires.
Attention : les ternaires sont complexes. Vérifiez qu'elles sont vraiment nécessaires avant d'en utiliser une.

Associations réflexives

  • Une entité associée à elle-même.
  • Exemple : SUPERVISER (0,N — 0,1) sur EMPLOYE.
  • Lectures : un employé supervise 0 à N employés · un employé est supervisé par 0 ou 1 employé.
  • Autres cas : composants/sous-composants, catégories/sous-catégories, articles liés, relations familiales.
  • Précaution : bien définir le sens de la relation (qui supervise qui) pour éviter l'ambiguïté.

Cas d'étude — Bibliothèque (3 étapes)

Étape 1 — Entités de base

  • LIVRE (numero_livre, titre, isbn, annee_publication, nombre_pages)
  • AUTEUR (numero_auteur, nom, prenom, date_naissance, nationalite)
  • ADHERENT (numero_adherent, nom, prenom, adresse, email, date_inscription)

Étape 2 — Associations simples

  • ECRIRE (1,N — 1,N) entre LIVRE et AUTEUR.
  • EMPRUNTER (0,N — 0,N) entre ADHERENT et LIVRE, avec attributs : date_emprunt, date_retour_prevue, date_retour_effective.
  • Problème identifié : un livre = un seul exemplaire ? Historique des emprunts ?

Étape 3 — Enrichissement

  • Ajout entité EXEMPLAIRE (numero_exemplaire, etat, date_acquisition) reliée à LIVRE par ETRE_EXEMPLAIRE (1,N — 1,1).
  • EMPRUNTER porte alors sur EXEMPLAIRE (pas LIVRE directement).
  • Ajout CATEGORIE (code_categorie, nom_categorie, description) reliée à LIVRE par APPARTENIR (1,N — 0,N).

Modèle complet

  • Entités : EXEMPLAIRE, AUTEUR, LIVRE, ADHERENT, CATEGORIE.
  • Associations : ETRE_EXEMPLAIRE, ECRIRE, EMPRUNTER, APPARTENIR.

Normalisation du MCD

  • Éviter la redondance, assurer la cohérence, faciliter la maintenance.
  • Atomicité : attributs non décomposables.
  • Unicité : chaque information n'apparaît qu'une fois.
  • Dépendance fonctionnelle : attributs dépendent de la clé entière.

Choix des identifiants — technique vs métier

  • Métier (risqué) : ex. code_barre seul → peut changer, dépend de règles externes.
  • Technique (recommandé) : ex. id_produit stable + garder code_barre comme attribut normal.
  • Avantages technique : stabilité (ne change jamais) · performance (numérique) · indépendance (pas de logique métier) · simplicité (géré par le système).

Erreurs courantes

Erreur 1 — Confondre entité et attribut

  • Mauvais : adresse comme simple attribut de PERSONNE si elle a plusieurs propriétés propres.
  • Bon : entité ADRESSE (id_adresse, rue, ville, code_postal) + association HABITER vers PERSONNE.
  • Quand créer une entité : l'objet a plusieurs attributs propres · il peut être référencé par d'autres entités · il a une existence indépendante.

Erreur 2 — Mauvaise analyse des cardinalités

  • Se poser les bonnes questions métier · penser aux cas limites · valider avec les utilisateurs · documenter.
  • Ex. : un client peut-il ne passer aucune commande ? → cardinalité 0,N côté CLIENT.

Erreur 3 — Identifiants instables

  • Email, téléphone, nom/prénom, numéro de compte bancaire → tous peuvent changer ou créer des doublons.
  • Solution : ajouter un num_personne stable et garder email/téléphone en attributs.

Problèmes d'identification fréquents

  • Identifiants composés trop complexes : plus de 3 attributs → revoir le modèle.
  • Confusion identifiant / référence externe : l'identifiant appartient à l'entité ; les références deviennent des clés étrangères au MLD.
  • Oubli de la minimalité : n'inclure que les attributs nécessaires, éviter la sur-identification.
  • Identifiants métier trop risqués : préférer les identifiants techniques et garder les métier comme attributs.

Conseils pour un MCD maintenable

  • Documenter les choix · Valider avec les utilisateurs · Anticiper les besoins futurs · Privilégier la simplicité · Respecter les conventions.

Vérifications finales

  • Identifiants stables ? Cardinalités cohérentes ? Pas de redondance inutile ?
  • Entités bien justifiées ? Associations sémantiquement correctes ?
Règle d'or : un bon MCD doit être compréhensible par les utilisateurs métier.

Récapitulatif — bénéfices d'un bon MCD

  • Compréhension métier partagée · Base solide pour MLD/MPD · Maintenance facilitée · Performance optimisée · Cohérence des données.
  • Étapes suivantes : transformation MCD → MLD · optimisation pour le SGBD · implémentation physique.
