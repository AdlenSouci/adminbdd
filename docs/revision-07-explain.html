EXPLAIN — Analyse des plans d'exécution

Tout le cours (explain.txt) — MySQL et PostgreSQL, toutes les colonnes, exemples détaillés.

À quoi sert EXPLAIN ?

  • Comprendre comment le moteur planifie et exécute une requête.
  • Identifier les problèmes de performance et les optimiser.

Scénarios typiques

  • Requêtes lentes : analyser le plan pour trouver le goulot d'étranglement.
  • Création / ajustement d'index : vérifier si les index existants sont utilisés.
  • Optimisation des jointures : comprendre l'ordre d'exécution et la méthode (hachage, boucle imbriquée…).
  • Sous-requêtes complexes : détecter si une sous-requête est exécutée plusieurs fois.

EXPLAIN en MySQL — Syntaxe

EXPLAIN [EXTENDED] [PARTITIONS] SELECT ...;
EXPLAIN EXTENDED SELECT ...;  -- puis SHOW WARNINGS pour voir les réécritures
EXPLAIN FORMAT=JSON SELECT ...;  -- format JSON, idéal pour analyses complexes

MySQL — Colonnes du résultat

ColonneDescription
idIdentifiant unique de l'étape. Chaque sous-requête reçoit un id différent.
select_typeType de la requête (voir ci-dessous)
tableTable (ou alias) utilisée dans cette étape
typeMéthode d'accès aux lignes — reflète l'efficacité (voir ci-dessous)
possible_keysIndex que MySQL pourrait utiliser (ou NULL)
keyIndex effectivement utilisé (ou NULL)
key_lenLongueur de l'index utilisé en octets (combien de colonnes)
refColonnes/valeurs utilisées pour associer les tables en jointure
rowsEstimation du nombre de lignes à examiner
ExtraInformations supplémentaires (voir ci-dessous)

MySQL — Valeurs de select_type

  • SIMPLE : requête simple, sans sous-requêtes ni unions.
  • PRIMARY : requête principale dans le cas de sous-requêtes.
  • SUBQUERY : sous-requête dans WHERE, FROM ou SELECT.
  • DERIVED : sous-requête dans la clause FROM.
  • UNION : requête faisant partie d'une union.
  • DEPENDENT SUBQUERY : sous-requête corrélée (dépend d'une requête externe).
  • UNCACHEABLE SUBQUERY : sous-requête qui ne peut pas être mise en cache.

MySQL — Valeurs de type (du pire au meilleur)

ValeurSignification
ALLScan complet de la table. Très inefficace sur grandes tables.
indexScan complet de l'index (plus rapide que ALL mais pas de filtre).
rangeRecherche dans un intervalle via index (BETWEEN, >, <). Efficace.
refUtilisation d'un index non-unique pour trouver des lignes.
eq_refIndex unique avec correspondance 1:1 (PK ou UNIQUE en jointure). Très efficace.
constAccès par constante, une seule ligne retournée. Très rapide.
systemTable à une seule ligne. Cas rare, similaire à const.
Surveiller en priorité : ALL (scan complet) avec un gros rows → créer un index.

MySQL — Valeurs de Extra

  • Using where : filtres appliqués après l'accès aux données.
  • Using index : accès via l'index uniquement, sans lire la table (index de couverture).
  • Using temporary : table temporaire pour résultats intermédiaires (GROUP BY, ORDER BY complexes).
  • Using filesort : tri externe (ORDER BY sans index adapté). À optimiser.
  • Using join buffer : tampon de jointure utilisé (Nested Loop sans index).
  • Using index condition : conditions WHERE appliquées directement via l'index (ICP).
  • No matching min/max row : aucune ligne ne correspond, peut être optimisé.

Exemple détaillé MySQL — Avant optimisation

SELECT e.emp_no, e.first_name, s.salary
FROM employees e
JOIN salaries s ON e.emp_no = s.emp_no
WHERE s.salary > 50000
  AND CURRENT_DATE BETWEEN s.from_date AND s.to_date;

Résultat EXPLAIN

tabletypekeyrowsExtra
s (salaries)ALLNULL299 379Using where
e (employees)eq_refPRIMARY1—

Problèmes identifiés

  • Scan complet de salaries (type = ALL) : MySQL parcourt toutes les lignes.
  • Aucun index utilisé pour le filtre sur salary et les dates (key = NULL).

Solution : créer un index couvrant

CREATE INDEX idx_salary_dates ON salaries (salary, from_date, to_date);
ANALYZE TABLE salaries;  -- mettre les statistiques à jour

Résultat après optimisation

tabletypekeyrowsExtra
salariesrangeidx_salary_dates~500Using index condition
employeeseq_refPRIMARY1—
type passe de ALL à range, rows de 299 379 à ~500 : gain massif.

EXPLAIN en PostgreSQL — Syntaxe et options

EXPLAIN [ (options) ] SELECT ...;
OptionDescription
ANALYZEExécute réellement la requête et mesure les temps réels.
VERBOSEDétails supplémentaires sur les nœuds du plan.
BUFFERSAffiche l'utilisation des tampons mémoire (cache disque).
COSTSActive/désactive l'affichage des coûts estimés.
TIMINGInclut/exclut les mesures de temps d'exécution.
EXPLAIN (ANALYZE, BUFFERS) SELECT ...;  -- exécute + mesures réelles + tampons

PostgreSQL — Nœuds du plan d'exécution

NœudDescription
Seq ScanParcours séquentiel de toute la table.
Index ScanUtilisation d'un index pour trouver des lignes spécifiques.
Bitmap Heap ScanLecture via un bitmap pour regrouper les lectures disque.
Hash JoinJointure par table de hachage (performant sur grandes données).
Nested LoopJointure par boucle imbriquée (inefficace sur grandes tables).
Merge JoinJointure par fusion (données déjà triées).
GatherAgrège les résultats de travailleurs parallèles.
Parallel Hash JoinJointure par hachage en parallèle.
Parallel Seq ScanScan séquentiel parallèle (charge répartie sur plusieurs workers).

Exemple PostgreSQL détaillé (cours)

  • Gather : agrège les résultats parallèles. Workers Planned: 2, Workers Launched: 2.
  • Parallel Hash Join : jointure salaries ↔ employees via hachage sur emp_no.
  • Parallel Seq Scan on salaries : filtre salary > 50000 AND CURRENT_DATE BETWEEN from_date AND to_date. Rows Removed by Filter: 874 744.
  • Parallel Hash : création table de hachage pour employees (18 Mo mémoire).
  • Parallel Seq Scan on employees : 100 008 lignes traitées par worker.
  • Planning Time: 1.014 ms · Execution Time: 143.514 ms.

Interprétation

  • Scans séquentiels parallèles : PostgreSQL répartit le travail (gain de temps).
  • 874 744 lignes rejetées par le filtre → index sur salary/dates améliorerait.
  • Accès disque (read) vs cache (shared hit) : le cache améliore la perf globale.

Suggestions optimisation PostgreSQL

CREATE INDEX idx_salary_date ON salaries (salary, from_date, to_date);
ANALYZE salaries;
ANALYZE employees;

Exercices du cours

  • Exercice 2 : comparer INNER JOIN vs LEFT JOIN — plan d'exécution, performances, cas métier de chaque type.

Conclusion

  • EXPLAIN est fondamental pour diagnostiquer et améliorer les performances SQL.
  • MySQL : base solide pour l'analyse des index et jointures.
  • PostgreSQL : analyse avancée avec mesures temps réel et statistiques détaillées.
  • Colonnes à surveiller en priorité : type (ALL = problème) et Extra (filesort, temporary = optimisation possible).
