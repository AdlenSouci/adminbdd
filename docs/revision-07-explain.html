EXPLAIN (MySQL & PostgreSQL)

Comprendre le plan d’exécution pour optimiser requêtes et index (support explain.txt).

À quoi ça sert ?

  • Requête lente : voir où le temps part (scan, jointure, tri).
  • Vérifier si les index existants sont utilisés.
  • Analyser l’ordre et la méthode des jointures.
  • Sous-requêtes : exécution répétée ou optimisée en une passe.

MySQL — syntaxe

  • EXPLAIN [EXTENDED] [PARTITIONS] SELECT …
  • EXPLAIN FORMAT=JSON SELECT … pour analyse détaillée / outils.
  • EXPLAIN EXTENDED + SHOW WARNINGS pour voir les réécritures de l’optimiseur.

MySQL — colonnes utiles du résultat

  • id : ordre / regroupement des étapes ; sous-requêtes ont des ids différents.
  • select_type : SIMPLE, PRIMARY, SUBQUERY, DERIVED (FROM), UNION, DEPENDENT SUBQUERY, etc.
  • table : table ou alias concerné.
  • type : qualité d’accès aux lignes (voir carte ci-dessous).
  • possible_keys / key : index candidats vs index réellement utilisé.
  • rows : estimation du nombre de lignes examinées.
  • Extra : Using where, Using index (index seul), Using temporary, Using filesort, Using index condition, etc.

MySQL — type (du pire au mieux, rappel cours)

  • ALL : scan complet table — à éviter sur grosses tables.
  • index : scan complet d’un index.
  • range : plage sur index (>, BETWEEN, etc.).
  • ref : accès index non unique.
  • eq_ref : une ligne par clé (souvent PK / unique en jointure).
  • const / system : accès direct constant (très bon).
Surveiller surtout type et Extra : ALL + gros rows ou Using filesort / Using temporary = pistes d’optimisation prioritaires.

Exemple cours (salaries / employees)

  • Avant index : salaries en type = ALL, key = NULL malgré PRIMARY en possible_keys → filtre sur salaire et dates sans index adapté.
  • Action : index du type (salary, from_date, to_date) ; ANALYZE TABLE salaries si stats obsolètes.
  • Après : type = range, index utilisé, Using index condition sur salaries.

PostgreSQL — syntaxe

  • EXPLAIN [ (option [, …]) ] SELECT …
  • EXPLAIN (ANALYZE, BUFFERS) : exécute vraiment la requête et affiche temps réel et accès disque/mémoire.
  • Autres options : VERBOSE, COSTS, TIMING, etc.

PostgreSQL — nœuds courants

  • Seq Scan : parcours séquentiel de toute la table.
  • Index Scan / Bitmap Index Scan : lecture via index (bitmap regroupe les accès heap).
  • Hash Join / Nested Loop / Merge Join : stratégies de jointure.
