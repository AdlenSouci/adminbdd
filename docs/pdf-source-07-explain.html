Analyse des Performances avec EXPLAIN (MySQL et PostgreSQL)
La commande EXPLAIN est une pierre angulaire de l’optimisation SQL. Elle permet de comprendre comment le moteur de base de données planifie et exécute une requête. En analysant ces plans d'exécution, les développeurs et administrateurs peuvent identifier les problèmes de performance et les optimiser efficacement.

Pourquoi et Quand Utiliser EXPLAIN ?
Scénarios typiques
Requêtes lentes :

Une requête qui prend plus de temps que prévu peut être optimisée en analysant son plan d'exécution.
Création ou ajustement d'index :

Vérifier si les index existants sont utilisés ou s'il est nécessaire d’en créer de nouveaux.
Optimisation des jointures :

Comprendre l’ordre d’exécution des jointures et leur méthode (par hachage, boucle imbriquée, etc.).
Sous-requêtes imbriquées ou complexes :

Identifier si une sous-requête est exécutée plusieurs fois ou si elle est optimisée par le moteur.
EXPLAIN en MySQL
Syntaxe
EXPLAIN [EXTENDED] [PARTITIONS] SELECT ...;
Variantes
EXPLAIN EXTENDED :

Ajoute des détails supplémentaires sur la transformation de la requête par l'optimiseur SQL. Utilisez ensuite SHOW WARNINGS pour voir ces informations.
EXPLAIN FORMAT=JSON :

Retourne un plan d'exécution au format JSON, idéal pour les analyses complexes.
Le résultat de la commande EXPLAIN est le suivant :

id : L'identifiant unique de l'étape dans le plan d'exécution. Chaque sous-requête ou étape complexe reçoit un identifiant différent.

Valeurs possibles : Un numéro entier qui identifie chaque étape ou sous-requête dans le plan d'exécution.
select_type : Indique le type de la requête ou de l'opération qui est en cours d'exécution. Cette colonne donne des informations sur la structure de la requête (par exemple, si c'est une requête simple, une sous-requête, ou une jointure).

Valeurs possibles :
SIMPLE : Requête simple, sans sous-requêtes ni unions.
PRIMARY : Requête principale dans le cas de sous-requêtes.
SUBQUERY : Sous-requête utilisée dans une clause WHERE, FROM, ou SELECT.
DERIVED : Sous-requête dans la clause FROM.
UNION : Requête faisant partie d’une union.
DEPENDENT SUBQUERY : Sous-requête corrélée.
UNCACHEABLE SUBQUERY : Sous-requête qui ne peut pas être mise en cache.
table : Le nom de la table utilisée dans l'étape spécifique de l'exécution de la requête.

Valeurs possibles : Le nom de la table concernée par l'étape ou sous-requête. Cela peut aussi être le nom d’une sous-requête dans le cas d’une requête dérivée.
type : Le type de jointure ou la méthode d'accès aux lignes dans la table. Il reflète l'efficacité de l'accès aux données.

Valeurs possibles (de la moins efficace à la plus efficace) :
ALL : Scan complet de la table. Inefficace, surtout pour les grandes tables
index : Scan complet de l'index sans filtrer (plus rapide qu'un scan complet de la table).
range : Recherche d'un intervalle de valeurs dans un index (efficace pour des requêtes avec des comparateurs comme BETWEEN, >, <).
ref : Utilisation d’un index pour trouver des lignes spécifiques.
eq_ref : Utilisation d'un index avec une correspondance 1:1 (très efficace). C'est en général un accès par clé primaire ou unique. Une seule ligne est retournée.
const : Une constante est utilisée, accès très rapide. Une seule ligne est retournée (très efficace).
system : Table contenant une seule ligne (cas rare, similaire à const).
possible_keys : Liste des index que MySQL pourrait utiliser pour exécuter la requête.

Valeurs possibles : Le nom des index disponibles pour cette table ou NULL si aucun index n'est disponible (le nom peut être PRIMARY si la clé primaire est utilisée)
key : L'index effectivement utilisé pour cette étape du plan d'exécution.

Valeurs possibles : Le nom de l'index utilisé, ou NULL si aucun index n'est utilisé.
key_len : La longueur de l'index utilisé, exprimée en octets. Cela montre combien de colonnes de l'index sont utilisées pour accéder aux données.

Valeurs possibles : La longueur de l'index utilisé, généralement en octets, ou NULL si aucun index n'est utilisé.
ref : Les colonnes ou valeurs utilisées pour associer les tables, généralement lors des jointures. Elle montre quelle colonne ou combinaison de colonnes est utilisée dans la condition de jointure.

Valeurs possibles : Les colonnes de la table utilisées dans la jointure, ou NULL si aucune colonne n'est impliquée dans la jointure.
rows : Le nombre estimé de lignes que MySQL doit examiner pour cette étape. C'est une estimation basée sur les statistiques de la table.

Valeurs possibles : Un nombre entier représentant le nombre de lignes estimé, ou NULL si l'estimation est inconnue.
Extra : Informations supplémentaires concernant l'exécution de la requête. Cette colonne fournit des détails sur l'utilisation de certaines techniques d'optimisation.

Valeurs possibles :
Using where : Indique que des filtres sont appliqués après l'accès aux données.
Using index : Montre que l'accès aux données a été effectué directement via l'index sans avoir besoin de scanner la table.
Using temporary : Utilisation d'une table temporaire pour stocker les résultats intermédiaires.
Using filesort : Tri externe utilisé, généralement pour les requêtes avec des ordres complexes.
Using join buffer : Utilisation d’un tampon de jointure pour exécuter la jointure.
Using index condition : Les conditions de la clause WHERE sont appliquées directement via l'index.
No matching min/max row : Aucune ligne ne correspond aux conditions de la requête, ce qui peut être optimisé.
L’analyse du plan d'exécution à l’aide de la commande EXPLAIN est essentielle pour comprendre comment MySQL exécute une requête et identifier les domaines où des optimisations peuvent être apportées. En particulier, les colonnes type et Extra sont cruciales pour identifier les points de ralentissement, tels que les scans complets de table ou l'absence d'utilisation d'index. En combinant les informations de ces colonnes avec des stratégies d'indexation appropriées, vous pouvez considérablement améliorer la performance de vos requêtes.

Exemple détaillé en MySQL
Requête initiale
SELECT e.emp_no, e.first_name, s.salary
FROM employees e
JOIN salaries s ON e.emp_no = s.emp_no
WHERE s.salary > 50000 AND CURRENT_DATE BETWEEN s.from_date AND s.to_date;

Problème identifié
Scan complet de la table salaries (type = ALL) :

Cela signifie que MySQL parcourt toutes les lignes de salaries pour appliquer le filtre s.salary > 50000 et la condition de temporalité.
Cette approche est inefficace, surtout pour des tables contenant un grand nombre de lignes.
Index non utilisé sur salaries :

Bien que PRIMARY soit listé comme un index possible, aucun index n’est réellement utilisé (key = NULL).
Cela peut être dû à l’absence d’un index couvrant la condition sur salary et la plage de dates.
Suggestions pour l’optimisation
Créer un index sur salary et les colonnes de temporalité (from_date, to_date) :

CREATE INDEX idx_salary_dates ON salaries (salary, from_date, to_date);
Cela permettra d’améliorer la recherche sur la combinaison de salary > 50000 et de la plage de dates.

Revoir l’ordre des jointures (si pertinent) : Dans ce cas précis, l’ordre des jointures semble optimal car salaries est analysée avant employees.

Vérifier les statistiques de la table : Si l’index existe déjà, I lest possible de mettre à jour les statistiques pour permettre à MySQL d’utiliser les index efficacement :

ANALYZE TABLE salaries;
Plan d'exécution attendu après optimisation
Après avoir créé un index approprié, le plan d'exécution devrait ressembler à ceci :

id select_type table type possible_keys key key_len ref rows Extra
1 SIMPLE salaries range PRIMARY, idx_salary_dates idx_salary_dates 8 NULL 500 Using index condition
1 SIMPLE employees eq_ref PRIMARY PRIMARY 4 salaries.emp_no 1 NULL
Explications des changements
type = range pour salaries :

L'index idx_salary_dates est utilisé pour parcourir uniquement les lignes correspondant au filtre s.salary > 50000 et à la plage de dates.
Extra = Using index condition :

Les conditions du WHERE sont appliquées directement via l’index, réduisant la charge.
EXPLAIN en PostgreSQL
Syntaxe
EXPLAIN [ (options) ] SELECT ...;
Options principales
Option Description
ANALYZE Exécute réellement la requête et mesure les temps réels pour chaque étape du plan d'exécution.
VERBOSE Fournit des détails supplémentaires sur les nœuds du plan.
BUFFERS Montre l’utilisation des tampons mémoire (cache disque).
COSTS Active ou désactive l’affichage des coûts estimés.
TIMING Inclut ou exclut les mesures de temps d'exécution.
Détails des Nœuds dans PostgreSQL
Nœud Description
Seq Scan Parcours séquentiel de la table entière.
Index Scan Utilisation d’un index pour trouver des lignes spécifiques.
Bitmap Heap Scan Lecture via un bitmap pour regrouper les lectures disque.
Hash Join Jointure basée sur une table de hachage (performant pour les grandes données).
Nested Loop Jointure par boucle imbriquée (inefficace pour les grandes tables).

Interprétation générale
Scans parallèles : La requête utilise des scans séquentiels parallèles sur les tables salaries et employees, ce qui montre que PostgreSQL a réparti le travail entre plusieurs processus pour accélérer l'exécution.

Jointure par hachage parallèle : L'utilisation d'une jointure par hachage parallèle permet de traiter efficacement les grandes tables, mais cela dépend de la façon dont les données sont réparties.

Filtres appliqués : Le filtre sur salary > 50000 et les dates (CURRENT_DATE BETWEEN from_date AND to_date) a réduit un grand nombre de lignes. Cependant, un grand nombre de lignes (874 744) a été rejeté lors de l'application du filtre, ce qui montre qu'une grande partie des données ne correspondait pas aux critères.

Accès au disque : Bien que des blocs aient été lus depuis le disque (read), une partie importante des données a été lue depuis le cache (shared hit), ce qui améliore la performance globale de la requête.

Suggestions d'optimisation
Indexer les colonnes utilisées dans les filtres : Pour améliorer la recherche, vous pouvez envisager de créer un index combiné sur les colonnes salary, from_date, et to_date dans la table salaries. Cela aidera PostgreSQL à éviter des scans séquentiels complets et à accélérer l'application du filtre.

Exemple de création d'un index :

CREATE INDEX idx_salary_date ON salaries (salary, from_date, to_date);
Analyser les statistiques des tables : Si la base de données contient des mises à jour ou des insertions ré

centes, assurez-vous que les statistiques sont à jour, ce qui permet à PostgreSQL de générer des plans d'exécution optimaux :

ANALYZE salaries;
ANALYZE employees;
Optimiser les jointures et les scans : Si la table employees est assez grande, vous pourriez envisager d'ajouter des index supplémentaires pour accélérer l'exécution de la jointure.
En résumé, bien que la requête utilise une approche parallèle efficace, il existe des possibilités d'optimisation, en particulier en ce qui concerne l'indexation et la gestion des données dans les scans séquentiels.

Conclusion
EXPLAIN est un outil fondamental pour diagnostiquer et améliorer les performances SQL. MySQL offre une base solide pour l’analyse des index et des jointures, tandis que PostgreSQL propose une analyse avancée avec des mesures en temps réel et des statistiques détaillées. Maîtriser ces commandes est essentiel pour tout professionnel travaillant avec des bases de données.
