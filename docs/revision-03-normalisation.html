Normalisation des bases de données

Tout le cours PDF « Normalisation » — 1NF, 2NF, 3NF, BCNF, anomalies, décompositions.

Objectifs du cours

  • Comprendre les principales formes normales : 1NF, 2NF, 3NF, BCNF.
  • Identifier les anomalies dues à une mauvaise conception relationnelle.
  • Apprendre à décomposer un schéma pour garantir cohérence et absence de redondance.

Pourquoi normaliser ? — Les anomalies

  • Redondances : informations répétées inutilement (ex. département « Informatique » sur chaque ligne).
  • Incohérences : valeurs contradictoires pour un même fait.
  • Anomalie d'insertion : impossibilité d'ajouter une donnée partielle.
  • Anomalie de modification : nécessité de mettre à jour plusieurs lignes.
  • Anomalie de suppression : perte accidentelle d'informations lors d'une suppression.
L'objectif : structurer les relations pour éliminer ces problèmes.

Exemple de départ (table non normalisée)

ÉtudiantCoursProfesseurDépartement
AliceBases de donnéesDupontInformatique
AliceRéseauxMartinInformatique
BobBases de donnéesDupontInformatique
  • Redondance du département « Informatique ».
  • Si le département change de nom → obligation de modifier plusieurs lignes.
  • Risque d'incohérences entre professeurs et leurs départements.

1NF — Première forme normale

Définition

  • Toutes les valeurs des attributs sont atomiques (indivisibles).
  • Il n'existe aucun groupe de valeurs répétées dans une même cellule.

Exemple non conforme

ÉtudiantCoursProfesseurs
AliceBD, RéseauxDupont, Martin
BobBDDupont

La colonne Cours contient plusieurs valeurs → viole la 1NF.

Exemple conforme (1NF atteinte)

ÉtudiantCoursProfesseur
AliceBDDupont
AliceRéseauxMartin
BobBDDupont

Chaque cellule contient une valeur unique et indivisible.

2NF — Deuxième forme normale

Définition

  • La table est en 1NF.
  • Chaque attribut non-clé dépend entièrement de la clé primaire (pas d'une partie seulement — pas de dépendance partielle).

Exemple non conforme (clé composée : Étudiant + Cours)

ÉtudiantCoursProfesseurDépartementVilleÉtudiant
AliceBDDupontInformatiqueParis
AliceRéseauxMartinInformatiqueParis
BobBDDupontInformatiqueLyon
Problème : VilleÉtudiant dépend seulement de Étudiant, pas de la clé entière (Étudiant, Cours) → dépendance partielle.

Décomposition en 2NF

  • Table Étudiant : Étudiant | VilleÉtudiant
  • Table Cours : Cours | Professeur | Département
  • Table Inscription : Étudiant | Cours (liaison pure)
Les dépendances partielles ont été supprimées.

3NF — Troisième forme normale

Définition

  • La table est en 2NF.
  • Aucun attribut non-clé ne dépend transitivement de la clé primaire.
  • Autrement dit : tout attribut non-clé dépend directement de la clé, pas d'un autre attribut non-clé.

Exemple non conforme (table Cours après 2NF)

CoursProfesseurDépartement
BDDupontInformatique
RéseauxMartinInformatique
  • Cours → Professeur
  • Professeur → Département
  • Donc Cours → Département (dépendance transitive) → viole la 3NF.

Décomposition en 3NF

  • Table Cours : Cours | Professeur
  • Table Professeur : Professeur | Département
Chaque attribut dépend désormais directement de la clé.

BCNF — Forme normale de Boyce-Codd

  • Pour chaque dépendance fonctionnelle X → Y, X est une clé candidate.
  • C'est une version plus stricte que la 3NF.
  • Une table en 3NF n'est pas forcément en BCNF (cas avec plusieurs clés candidates qui se chevauchent).

Récapitulatif des formes normales

FormeCondition supplémentaire
1NFValeurs atomiques, pas de groupes répétés
2NF1NF + pas de dépendance partielle de la clé composée
3NF2NF + pas de dépendance transitive
BCNFPour toute DF X→Y, X est clé candidate
