Questions Fréquentes (FAQ)

FAQ - Différence PRIMARY KEY vs UNIQUE

PRIMARY KEY ✅ Une seule par table ✅ Jamais NULL ✅ Index créé automatiquement ✅ Identifie chaque ligne UNIQUE ✅ Plusieurs par table ✅ Peut être NULL ✅ Index créé automatiquement ✅ Garantit l’unicité 💡 En résumé : PRIMARY KEY = UNIQUE + NOT NULL + une seule par table 1 CREATE TABLE users ( 2 id INT PRIMARY KEY, 3 email VARCHAR(100) 4 ); 1 CREATE TABLE users ( 2 id INT PRIMARY KEY, 3 email VARCHAR(100) UNIQUE 4 );

FAQ - Quand utiliser CASCADE ?

✅ Utilisez ON DELETE CASCADE quand : La relation est de type "composition" (l’enfant n’existe pas sans le parent) Exemple : Commande → Lignes de commande Exemple : Article → Commentaires 1 CREATE TABLE lignes_commande ( 2 id INTEGER PRIMARY KEY, 3 commande_id INTEGER, 4 FOREIGN KEY (commande_id) REFERENCES commandes(id) 5 ON DELETE CASCADE -- Si commande supprimée, supprimer ses lignes 6 );

❌ N’utilisez PAS CASCADE quand :
La relation est de type "référence" (l’enfant peut exister indépendamment)
Exemple : Commande → Client
Préférez ON DELETE RESTRICT ou SET NULL
1 CREATE TABLE commandes (
2 id INTEGER PRIMARY KEY,
3 client_id INTEGER,
4 FOREIGN KEY (client_id) REFERENCES clients(id)
5 ON DELETE RESTRICT -- Empêche de supprimer un client avec commandes
6 );

SERIAL (ancien) Problèmes : Crée une séquence séparée Séquence non supprimée avec DROP TABLE Pas standard SQL Peut insérer manuellement une valeur IDENTITY (moderne) ⭐ Avantages : Standard SQL:2003 Mieux intégré à la table ALWAYS empêche insertion manuelle Nettoyage automatique 1 CREATE TABLE users ( 2 id SERIAL PRIMARY KEY 3 ); 1 CREATE TABLE users ( 2 id INT GENERATED ALWAYS 3 AS IDENTITY PRIMARY KEY 4 );

Points clés à retenir

  • Toujours définir des clés primaires - Identifient de manière unique chaque ligne
  • Utiliser GENERATED AS IDENTITY - Standard SQL moderne pour l’auto-incrémentation
  • Ne pas oublier les clés étrangères - Garantissent l’intégrité référentielle
  • Nommer toutes les contraintes - Facilite la maintenance et le debugging
  • Choisir les bonnes actions référentielles - CASCADE pour composition, RESTRICT pour référence
  • Ajouter des contraintes CHECK - Validation des données au niveau base
  • Utiliser NOT NULL judicieusement - Rendre explicites les colonnes obligatoires
  • Penser aux performances - Les contraintes créent des index automatiques
  • Documenter votre schéma - Commentaires, diagrammes, migrations versionnées
  • Tester avant production - Vérifier toutes les contraintes avec des données réelles
