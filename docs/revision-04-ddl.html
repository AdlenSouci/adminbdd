DDL — Data Definition Language

Tout le cours PDF « DDL » — syntaxe, types, contraintes, FK, ALTER, DROP, FAQ, exercices.

Qu'est-ce que le DDL ?

  • Sous-ensemble SQL qui permet de définir et modifier la structure des objets de la base de données.
  • Modifie la structure, pas les données (ça c'est le DML).
  • CREATE — créer des objets (tables, vues, index…)
  • ALTER — modifier des objets existants
  • DROP — supprimer des objets
  • TRUNCATE — vider une table (structure conservée)

CREATE TABLE — squelette

CREATE TABLE nom_table (
  colonne1 TYPE [CONTRAINTES],
  colonne2 TYPE [CONTRAINTES],
  [CONTRAINTES_TABLE]
);

Exemple minimal

CREATE TABLE employes (
  id      INTEGER,
  nom     VARCHAR(100),
  salaire DECIMAL(10,2)
);

Clés Primaires (PRIMARY KEY)

  • Valeurs uniques · NON NULL · une seule PK par table · peut être composée.

3 syntaxes

-- 1. Niveau colonne
CREATE TABLE clients (id INTEGER PRIMARY KEY, nom VARCHAR(100));

-- 2. Niveau table
CREATE TABLE clients (id INTEGER, nom VARCHAR(100), PRIMARY KEY (id));

-- 3. Avec nom de contrainte
CREATE TABLE clients (
  id INTEGER, nom VARCHAR(100),
  CONSTRAINT pk_clients PRIMARY KEY (id)
);

Clé primaire composée

CREATE TABLE inscriptions (
  etudiant_id INTEGER,
  cours_id    INTEGER,
  PRIMARY KEY (etudiant_id, cours_id)
);

PostgreSQL — GENERATED AS IDENTITY

  • GENERATED ALWAYS : interdit d'insérer manuellement l'id. Protection maximale.
  • GENERATED BY DEFAULT : insertion manuelle possible (utile pour imports/migrations).

Options avancées IDENTITY

  • START WITH : valeur de départ · INCREMENT BY : pas (peut être négatif)
  • MINVALUE / MAXVALUE : limites · CACHE : pré-allocation mémoire (performance)
  • CYCLE : recommence au min si max atteint
Recommandation cours : préférez GENERATED AS IDENTITY (standard SQL).

Clés Étrangères (FOREIGN KEY)

  • Établit une relation entre deux tables ; garantit l'intégrité référentielle.
-- Table parent
CREATE TABLE clients (id INTEGER PRIMARY KEY, nom VARCHAR(100));

-- Table enfant
CREATE TABLE commandes (
  id        INTEGER PRIMARY KEY,
  client_id INTEGER REFERENCES clients(id),
  date_commande DATE
);

Syntaxes alternatives

-- Au niveau table
FOREIGN KEY (client_id) REFERENCES clients(id)

-- Avec nom de contrainte
CONSTRAINT fk_commandes_clients FOREIGN KEY (client_id) REFERENCES clients(id)

Actions référentielles ON DELETE / ON UPDATE

  • CASCADE : supprime/met à jour automatiquement les lignes enfants.
  • SET NULL : met la FK à NULL.
  • SET DEFAULT : met la valeur par défaut.
  • RESTRICT / NO ACTION : empêche la modification/suppression si des enfants existent (comportement par défaut).

FK composées (clé parent composée)

CREATE TABLE notes (
  etudiant_id INTEGER,
  cours_id    INTEGER,
  note        DECIMAL(4,2),
  FOREIGN KEY (etudiant_id, cours_id)
    REFERENCES inscriptions(etudiant_id, cours_id)
    ON DELETE CASCADE
);

Contraintes — vue d'ensemble

ContrainteDescription
PRIMARY KEYIdentifiant unique, non null
FOREIGN KEYRéférence à une autre table
UNIQUEValeurs uniques (NULL autorisé)
NOT NULLValeur obligatoire
CHECKCondition personnalisée
DEFAULTValeur par défaut si rien fourni

Contrainte UNIQUE

  • Garantit l'unicité des valeurs mais autorise NULL.
  • Plusieurs UNIQUE par table (contrairement à PK).
-- Colonne unique
CREATE TABLE utilisateurs (
  id     INTEGER PRIMARY KEY,
  email  VARCHAR(100) UNIQUE,
  pseudo VARCHAR(50) UNIQUE
);
-- Contrainte multi-colonnes
UNIQUE (salle_id, date_reservation, heure_debut)

Contrainte NOT NULL

  • Rend une colonne obligatoire. Insertion sans valeur → erreur.
nom   VARCHAR(100) NOT NULL,
prenom VARCHAR(100) NOT NULL

Contrainte CHECK

  • Définit une condition personnalisée que les données doivent respecter.
-- Simple
prix DECIMAL(10,2) CHECK (prix > 0),
stock INTEGER CHECK (stock >= 0),
age INTEGER CHECK (age >= 18 AND age <= 65)

-- Entre colonnes
CHECK (date_fin > date_debut)

-- Expression
CHECK (email LIKE '%@%.%')

-- Valeurs dans une liste
statut VARCHAR(20) CHECK (statut IN ('brouillon','publie','archive'))

Contrainte DEFAULT

  • Définit une valeur par défaut si aucune valeur n'est fournie à l'insertion.
date_creation TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
statut        VARCHAR(20) DEFAULT 'brouillon',
vues          INTEGER DEFAULT 0

-- PostgreSQL
DEFAULT CURRENT_DATE, DEFAULT current_user

-- MySQL spécifique
ON UPDATE CURRENT_TIMESTAMP  -- mise à jour auto du timestamp

Nommage des contraintes (convention)

  • pk_nomtable pour PRIMARY KEY
  • fk_table1_table2 pour FOREIGN KEY
  • uk_nomtable_colonne pour UNIQUE
  • chk_nomtable_description pour CHECK
Toujours nommer vos contraintes : facilite la maintenance et le débogage avec DROP CONSTRAINT.

ALTER TABLE

-- Ajouter colonne
ALTER TABLE employes ADD COLUMN telephone VARCHAR(20);
ALTER TABLE employes ADD COLUMN date_naissance DATE NOT NULL DEFAULT '1990-01-01';

-- Supprimer colonne
ALTER TABLE employes DROP COLUMN telephone;

-- Modifier type (PostgreSQL)
ALTER TABLE employes ALTER COLUMN nom TYPE VARCHAR(200);
ALTER TABLE employes ALTER COLUMN salaire SET NOT NULL;

-- Modifier type (MySQL)
ALTER TABLE employes MODIFY COLUMN nom VARCHAR(200);

-- Ajouter contrainte
ALTER TABLE employes ADD CONSTRAINT chk_age CHECK (age >= 18);
ALTER TABLE commandes ADD CONSTRAINT fk_client
  FOREIGN KEY (client_id) REFERENCES clients(id);

-- Supprimer contrainte
ALTER TABLE employes DROP CONSTRAINT chk_age;

-- Renommer table
ALTER TABLE employes RENAME TO personnel;

DROP TABLE et TRUNCATE

-- DROP : supprime table + données
DROP TABLE employes;
DROP TABLE IF EXISTS employes;
DROP TABLE employes CASCADE;  -- avec dépendances (PostgreSQL)

-- TRUNCATE : vide la table rapidement (structure conservée)
TRUNCATE TABLE employes;
TRUNCATE TABLE employes RESTART IDENTITY;  -- réinitialise séquences (PG)
TRUNCATE TABLE employes CASCADE;           -- avec tables liées

Bonnes pratiques DDL

À faire

  • Toujours nommer les contraintes.
  • Utiliser GENERATED AS IDENTITY (standard SQL).
  • Définir les FK avec actions ON DELETE/ON UPDATE appropriées.
  • Utiliser NOT NULL pour les colonnes obligatoires.
  • Ajouter des CHECK pour la validation métier.
  • Documenter le schéma.

À éviter

  • Oublier les clés étrangères · Pas de contrainte sur données critiques.
  • Types trop larges (VARCHAR(999)) · Ignorer l'indexation · Noms peu clairs.
  • Modifications de structure sans backup.

Checklist avant CREATE TABLE

  • Clé primaire définie (AUTO_INCREMENT / IDENTITY)
  • Toutes les clés étrangères identifiées
  • Contraintes UNIQUE nécessaires ajoutées
  • NOT NULL sur colonnes obligatoires
  • CHECK pour validation des données
  • Valeurs DEFAULT appropriées
  • Actions ON DELETE / ON UPDATE spécifiées
  • Toutes les contraintes nommées
  • Types de données vérifiés
  • Index pour les performances prévus

Résumé : PRIMARY KEY = UNIQUE + NOT NULL + une seule par table.

FAQ — Quand utiliser CASCADE ?

  • Oui — relation de composition (l'enfant n'a pas de sens sans le parent) : commande → lignes de commande, article → commentaires.
  • Non — relation de référence (l'enfant peut exister indépendamment) : commande → client → préférez RESTRICT ou SET NULL.

FAQ — SERIAL vs IDENTITY (PostgreSQL)

SERIAL (ancien)IDENTITY (recommandé)

SéquenceSéparée, pas supprimée avec DROP TABLEIntégrée, nettoyage auto
Insertion manuellePossibleALWAYS empêche, BY DEFAULT autorise

FAQ — Niveau colonne vs niveau table

  • Niveau colonne : concis pour contraintes simples.
  • Niveau table (recommandé) : noms explicites, obligatoire pour contraintes multi-colonnes, meilleure lisibilité.

FAQ — Performances des contraintes

  • Impact positif : PK et UNIQUE créent un index automatique → recherches rapides. FK garantit l'intégrité sans code applicatif. NOT NULL optimise le stockage.
  • À considérer : CHECK vérifié à chaque INSERT/UPDATE. FK vérifiée aussi sur DELETE. Trop d'index ralentissent les écritures.
Les contraintes sont essentielles. L'impact performance est généralement négligeable comparé aux bénéfices.

FAQ — Gérer les migrations

  • Toujours scripter les changements (fichiers SQL versionnés : migration_001_…sql).
  • Outils recommandés : Flyway (Java), Liquibase (multi-langage), Alembic (Python), Knex.js (JavaScript).

Points clés à retenir (10)

  • 1. Toujours définir des clés primaires.
  • 2. Utiliser GENERATED AS IDENTITY (standard SQL moderne).
  • 3. Ne pas oublier les clés étrangères.
  • 4. Nommer toutes les contraintes.
  • 5. Choisir les bonnes actions référentielles (CASCADE composition, RESTRICT référence).
  • 6. Ajouter des CHECK pour la validation des données.
  • 7. Utiliser NOT NULL sur les colonnes obligatoires.
  • 8. Penser aux performances (contraintes = index auto).
  • 9. Documenter le schéma (commentaires, diagrammes, migrations).
  • 10. Tester avant production avec des données réelles.
