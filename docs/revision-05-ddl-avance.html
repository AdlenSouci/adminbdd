  • Bases maintenables : simplifier les opérations d'administration.

Partitionnement — concept général

  • Diviser une grande table en partitions physiques tout en gardant une vue logique unique.
  • Le SGBD peut ignorer les partitions non pertinentes lors d'une requête : Partition Pruning.
  • Ex. : SELECT * FROM sales WHERE YEAR(sale_date) = 2023 → ne scanne que la partition 2023.

Stratégie 1 — Partitionnement par Plage (RANGE)

  • Idéal pour les données temporelles (logs, commandes par année/mois…).
  • La colonne de partitionnement doit faire partie de la clé primaire.
  • Partition pruning : le SGBD ignore les partitions non pertinentes.
CREATE TABLE sales (
    id        INT,
    sale_date DATE NOT NULL,
    PRIMARY KEY (id, sale_date)
)
PARTITION BY RANGE (YEAR(sale_date)) (
    PARTITION p2022 VALUES LESS THAN (2023),
    PARTITION p2023 VALUES LESS THAN (2024),
    PARTITION p_future VALUES LESS THAN MAXVALUE
);

Stratégie 2 — Partitionnement par Liste (LIST)

  • Parfait pour les données catégorielles : statut, pays, région.
  • Chaque partition reçoit un ensemble de valeurs explicites.
CREATE TABLE customers (
    id     INT,
    region VARCHAR(50) NOT NULL,
    PRIMARY KEY (id, region)
)
PARTITION BY LIST COLUMNS(region) (
    PARTITION p_north VALUES IN ('Hauts-de-France', 'Normandie'),
    PARTITION p_south VALUES IN ('PACA', 'Occitanie')
);

Stratégie 3 — Partitionnement par Hachage (HASH)

  • Répartit les données uniformément sur N partitions quand il n'y a pas de colonne de partitionnement logique.
  • Cas d'usage : équilibrer la charge d'écriture, éviter les « points chauds » sur une table très sollicitée.
  • Le SGBD calcule HASH(valeur) % N pour déterminer la partition de destination.
CREATE TABLE users (
    id       INT PRIMARY KEY,
    username VARCHAR(50)
)
PARTITION BY HASH(id)
PARTITIONS 4;

Index Composites

  • Index sur plusieurs colonnes. L'ordre est CRUCIAL.
  • Règle d'or : mettre la colonne la plus sélective en premier (celle qui filtre le plus de lignes).

Exemple cours

  • Requête : WHERE department_id = 10 AND status = 'active'
  • Mauvais index : (status, department_id) — status peu sélectif, peu efficace.
  • Bon index : (department_id, status) — department_id plus sélectif, réduit rapidement les lignes.
Un bon index peut diviser le temps d'une requête par 1000. Un mauvais index peut la ralentir.

Index de Couverture (Covering Index)

  • L'index contient toutes les colonnes requises par la requête (filtre ET projections).
  • Le SGBD répond sans jamais lire la table heap → 1 opération au lieu de 2.
  • C'est la forme d'optimisation de lecture la plus puissante.

Exemple cours

  • Requête : SELECT email, last_login FROM users WHERE username = 'jdoe'
  • Index simple (username) → trouve jdoe dans l'index, doit ensuite lire la table pour email et last_login (2 opérations).
  • Index couverture (username, email, last_login) → tout dans l'index, ne lit jamais la table (1 opération).
