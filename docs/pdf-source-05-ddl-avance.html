Stratégie 1 :
Partitionnement par Plage
(RANGE)
**Clé :** La colonne de partitionnement (`sale_date`)
**doit** faire partie de la clé primaire.

Table Logique: sales

p2022 p2023 p2024
Le SGBD ignore les partitions non pertinentes ("Partition Pruning").

CREATE TABLE sales (
id INT,
sale_date DATE NOT NULL,
PRIMARY KEY (id, sale_date)
)
PARTITION BY RANGE (YEAR(sale_date)) (
PARTITION p2022 VALUES LESS THAN (2023),
PARTITION p2023 VALUES LESS THAN (2024),
PARTITION p_future VALUES LESS THAN MAXVALUE
);

Index Composites
Requête type : SELECT * FROM employees WHERE department_id = 10 AND status = 'active';
Mauvais index : (status, department_id)
status est peu sélectif. L’index est peu efficace.
Bon index : (department_id, status)
department_id est plus sélectif. L’index réduit rapidement le nombre de lignes à scanner.
Règle d’or : Mettez la colonne la plus sélective en premier.
U n i n d e x s u r p l u s i e u r s c o l o n n e s . L ’ o r d r e e s t C R U C I A L .

Index de Couverture
Un index qui contient toutes les colonnes requises par une requête.
Requête type : SELECT email, last_login FROM users WHERE username = 'jdoe';
Index simple : (username)
Le SGBD trouve 'jdoe' dans l’index, puis doit aller lire la table pour récupérer email et last_login. (2
opérations)
Index de couverture : (username, email, last_login)
Le SGBD trouve tout ce dont il a besoin dans l’index. Il ne lit jamais la table. (1 opération)
C’est la forme d’optimisation de lecture la plus puissante.

Optimisation pour OLTP vs OLAP
OLTP (OnLine Transaction Processing)
Quoi : Systèmes du quotidien (e-commerce, banque).
Charge : Beaucoup de petites écritures rapides.
Optimisation : Normalisation, index très ciblés.
OLAP (OnLine Analytical Processing)
Quoi : Systèmes d’analyse (Data Warehouses).
Charge : Requêtes complexes qui lisent des millions de lignes.
Optimisation : Dénormalisation, partitionnement, vues matérialisées.
